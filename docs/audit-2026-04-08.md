# Audit Report — 2026-04-08

**Project:** PITH v0.1.0
**Scope:** Security + Performance (CLI tool — no accessibility audit applicable)
**Auditor:** Claude Code automated audit

---

## Security Findings

### P0 (Critical)

#### S1. License HMAC uses symmetric secret embedded in source code
**File:** `pith/license/__init__.py:27`
```python
_LICENSE_SALT = b"pithhq-license-salt-v1"
```
The HMAC salt is hardcoded in the open-source codebase. Any attacker can read it, forge a valid `license.json` with any tier, and bypass the license system entirely. The license key format prefix (`pithhq_`) is also public.

**Impact:** Complete license bypass — anyone can generate a valid license file without paying.

**Remediation:**
1. Replace HMAC with **asymmetric signing** (Ed25519). Ship only the public key in the binary; keep the private key on the server.
2. The activation endpoint should return a server-signed license blob.
3. This is already tracked as a TODO (lines 7, 26) — it is a **launch blocker**.

---

#### S2. Machine fingerprint is trivially spoofable
**File:** `pith/license/__init__.py:48-50`
```python
raw = f"{socket.gethostname()}{platform.platform()}{platform.processor()}"
return hashlib.sha256(raw.encode()).hexdigest()[:32]
```
The fingerprint uses hostname + platform string + processor name — all user-controllable. An attacker can copy a `license.json` to another machine and set the hostname/platform to match.

**Impact:** License transferred between machines without limit, defeating per-machine licensing.

**Remediation:**
- Add hardware-unique identifiers: MAC address (`uuid.getnode()`), disk serial, or OS-level machine ID (`/etc/machine-id` on Linux, `IOPlatformUUID` on macOS, Windows `MachineGuid` registry key).
- Combine multiple signals so spoofing requires root access.

### P1 (High)

#### S3. Lemon Squeezy activation has no Authorization header
**File:** `pith/license/__init__.py:81-86`
```python
# TODO: add Lemon Squeezy API key header (Authorization: Bearer ...)
response = httpx.post(
    "https://api.lemonsqueezy.com/v1/licenses/validate",
    json={"license_key": key},
    timeout=30,
)
```
The API call will fail or be unauthenticated. This is a TODO but worth flagging as P1 since activation is non-functional without it.

**Remediation:** Add the Authorization header using an env var (never hardcoded). The Lemon Squeezy API key should be bundled in the binary via build-time embedding, not user-provided.

---

#### S4. Scheduler writes executable config without restrictive permissions
**File:** `pith/init/scheduler.py:28, 39, 65, 106`

Systemd units, launchd plists, and Task Scheduler XML are written with default umask permissions. On multi-user systems, other users could modify the scheduler files to execute arbitrary commands.

**Remediation:** Set file permissions to `0o600` after writing:
```python
service_path.write_text(...)
service_path.chmod(0o600)
```

---

#### S5. License file written with default permissions
**File:** `pith/license/__init__.py:107`
```python
license_path.write_text(json.dumps(license_data, indent=2), encoding="utf-8")
```
The license file contains the license key in plaintext. Default permissions may allow other users to read it.

**Remediation:** `license_path.chmod(0o600)` after writing.

---

#### S6. Schema name not validated for path traversal
**File:** `pith/schema/__init__.py:167`
```python
schema_dir = schemas_root / name
```
If `name` contains `../`, this could read files outside the schemas directory. The `name` comes from `pith.config.json` which is user-controlled.

**Remediation:**
```python
schema_dir = (schemas_root / name).resolve()
if not schema_dir.is_relative_to(schemas_root.resolve()):
    raise SchemaValidationError(name, "schema name contains path traversal")
```

---

#### S7. Ingest output path not validated against vault boundary
**File:** `pith/ingest/__init__.py:89-90`
```python
def _page_path_for(source: Path, vault_path: Path) -> Path:
    return vault_path / f"{source.stem}.md"
```
While `Path.stem` strips directory components, there is no validation that the resulting path stays within the vault directory. Unusual filenames could produce unexpected stems.

**Remediation:** Add a `resolve()` check:
```python
page_path = (vault_path / f"{source.stem}.md").resolve()
if not page_path.is_relative_to(vault_path.resolve()):
    raise ValueError(f"output path escapes vault: {page_path}")
```

---

#### S8. Ollama base_url accepts any host — SSRF risk in Tauri GUI
**Files:** `pith/config/models.py:70-76`, `pith/ingest/__init__.py:113`, `pith/query/__init__.py:82`

The Ollama `base_url` is validated as a URL but accepts any host. A malicious config could point this at cloud metadata endpoints (e.g., `http://169.254.169.254/`) or internal services. The application POSTs document content to whatever URL is configured.

**Impact:** Low in CLI context (user controls their own config). Becomes P1 when Tauri GUI ships with shared or imported configs.

**Remediation:** Restrict to localhost/private ranges by default; add explicit flag for non-local URLs.

### P2 (Medium)

#### S9. No input size limits on parsed files
**Files:** `pith/parsers/text.py:79`, `pith/parsers/docx.py:40`, `pith/parsers/pdf.py:116`

All parsers load entire files into memory without size checks. A maliciously large file (e.g., 10GB text file) would cause OOM.

**Remediation:** Add a configurable max file size check before parsing (e.g., 100MB default).

---

#### S10. YAML frontmatter parsing uses yaml.safe_load (good) but no size limit
**File:** `pith/lint/__init__.py:50`, `pith/schema/__init__.py:180`

`yaml.safe_load` is used correctly (not `yaml.load`), which prevents arbitrary code execution. However, deeply nested YAML could cause excessive memory usage.

**Impact:** Low — files are local and user-controlled. Defensive, not exploitable remotely.

---

#### S11. Frontmatter parsing inconsistent between modules
**Files:** `pith/export/__init__.py:39-60` vs `pith/lint/__init__.py:44-53`

Export uses a hand-rolled `key:value` parser for frontmatter; lint uses `yaml.safe_load`. Different parsing logic can produce different results for the same frontmatter, leading to subtle bugs (e.g., a page flagged stale by lint but exported with wrong metadata).

**Remediation:** Extract frontmatter parsing into a shared utility using `yaml.safe_load` consistently.

---

#### S12. `pith init` silently overwrites existing config
**File:** `pith/init/__init__.py:156-160`

The init wizard writes `pith.config.json` without checking if one already exists, silently overwriting configuration.

**Remediation:** Check for existing file and prompt for confirmation before overwriting.

---

#### S13. Broad exception handling suppresses errors
**Files:** `pith/license/__init__.py:139`, `pith/cli/__init__.py:137`
```python
except Exception:  # noqa: BLE001
```
Broad exception handling during license validation and activation could hide unexpected errors. The `noqa` comment shows awareness but the pattern is still risky.

**Remediation:** Log the exception at debug level before returning the invalid result.

## Performance Findings

### Critical

#### P1. New httpx.AsyncClient created per API call — no connection reuse
**Files:** `pith/ingest/__init__.py:112, 147`
```python
async with httpx.AsyncClient(timeout=120.0) as client:
    response = await client.post(...)
```
Each chunk creates a new `AsyncClient`, establishing a new TCP+TLS connection. For a document with 50 chunks, that's 50 TLS handshakes to the Anthropic API.

**Impact:** Ingest of large documents is significantly slower than necessary. TLS handshake overhead (~100-300ms per connection) adds up.

**Remediation:** Create the client once in `ingest_file()` and pass it to `_call_anthropic`/`_call_ollama`:
```python
async with httpx.AsyncClient(timeout=120.0) as client:
    for chunk in chunks:
        section_md = await _call_anthropic(chunk, model, api_key, system_prompt, client)
```

---

#### P2. Chunks processed sequentially — no concurrency
**File:** `pith/ingest/__init__.py:216-229`

Chunks are sent to the API one at a time in a `for` loop. With 120s timeout per call, a 50-chunk document could take up to 100 minutes if each call takes 2 minutes.

**Impact:** Ingest is CPU-idle while waiting for API responses. Could be 3-5x faster with concurrency.

**Remediation:** Use `asyncio.Semaphore` to process chunks concurrently (e.g., 5 at a time):
```python
sem = asyncio.Semaphore(5)
async def process_chunk(chunk):
    async with sem:
        return await _call_anthropic(chunk, model, api_key, system_prompt, client)
tasks = [process_chunk(c) for c in chunks]
results = await asyncio.gather(*tasks, return_exceptions=True)
```

### Opportunities

#### P3. `_ocr_page` renders PDF page image twice (bug)
**File:** `pith/parsers/pdf.py:71, 91`
```python
def _ocr_page(page: pdfplumber.pdf.Page) -> str:
    _set_tessdata_prefix()
    image = page.to_image().original  # line 71 — rendered, then overwritten
    ...
    image = page.to_image().original  # line 91 — rendered again
```
Line 71 renders the page to an image before the docstring (misplaced code), then line 91 renders it again inside the `try` block. Every OCR page is rasterized twice, wasting 200-500ms per page.

**Remediation:** Remove line 71 and move `_set_tessdata_prefix()` below the docstring.

---

#### P4. `load_config` called per OCR page — redundant disk reads
**File:** `pith/parsers/pdf.py:57-66`

`_ocr_language()` calls `load_config(Path("pith.config.json"))` for every page that triggers OCR. A 100-page scanned PDF causes 100 redundant config loads (file read + JSON parse + Pydantic validation).

**Remediation:** Pass `language: str | None` as a parameter from the caller, or cache with `functools.lru_cache`.

---

#### P5. CLI startup imports are well-structured
**File:** `pith/cli/__init__.py`

Heavy imports (`pith.ingest`, `pith.query`, `pith.export`, `pith.init`) are deferred inside command functions. `pith version` and `pith init` don't load API clients. Good pattern — no action needed.

---

#### P6. Text parser reads entire file into memory
**File:** `pith/parsers/text.py:79`
```python
text = path.read_text(encoding="utf-8")
```
For very large text files (>100MB), this loads everything into memory at once.

**Remediation (low priority):** For text files, consider streaming line-by-line for the heading detection pass. Unlikely to be an issue in practice since most documents are <10MB.

---

#### P7. PDF parser loads all pages before processing
**File:** `pith/parsers/pdf.py:126`

pdfplumber opens the entire PDF. For very large PDFs (1000+ pages), this could use significant memory. The page-by-page iteration is correct, but pdfplumber loads all pages into memory on open.

**Remediation (low priority):** For large PDFs, consider processing in page batches. pdfplumber's memory usage is generally acceptable for typical document sizes.

---

#### P8. Contradiction check sends all page pairs sequentially
**File:** `pith/lint/__init__.py:141-168`

The lint contradiction check sends page pairs to Ollama one at a time. A vault with many tagged pages could generate O(n^2) pairs.

**Remediation:** Add concurrency (similar to P2) and consider limiting the number of pairs checked (e.g., max 50 pairs per lint run).

---

#### P9. CSV export uses pandas for a simple operation
**File:** `pith/export/__init__.py:161`
```python
import pandas as pd
```
pandas is a heavy dependency (~50MB) imported just to write a CSV file. The stdlib `csv` module handles this trivially.

**Remediation:** Replace with stdlib `csv.DictWriter`:
```python
import csv
with output_path.open("w", newline="", encoding="utf-8") as f:
    writer = csv.DictWriter(f, fieldnames=["title", "source", ...])
    writer.writeheader()
    writer.writerows(rows)
```
This also eliminates the pandas-stubs mypy issue.

---

#### P10. Query context loading reads all vault files on every query
**File:** `pith/query/__init__.py:33-34`

Every `pith query` reads all `.md` files from disk. For a large vault (1000+ files), this adds latency to every query.

**Remediation (future):** Consider caching the concatenated context with a file-watcher or hash-based invalidation.
